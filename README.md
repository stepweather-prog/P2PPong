# P2PPong v4.2 — Слепой Улей + DHT

Инженерная панель для наблюдения за работой P2P-движка в реальном времени. Минимум UI, максимум прозрачности.

## Что это такое

Это не мессенджер. Это отладочный стенд для разработчика, аудитора или исследователя. Позволяет:

- Наблюдать за каждым этапом ECDH-рукопожатия
- Видеть полную таблицу DHT-корзин в реальном времени
- Вручную записывать и читать данные в DHT-сети
- Отслеживать fallback-логику (DHT → WebSocket)
- Проверять шифрование блобов на каждом шаге

## Структура панели

| Секция | Что показывает |
|--------|---------------|
| Статус | Онлайн/офлайн, Peer ID, счётчики каналов и DHT-хранилища |
| Маяки | Создание и отображение активных маяков с оставшимся временем |
| Подключиться | Поле ввода Peer ID + кнопка для ECDH-рукопожатия |
| Заявки | Входящие запросы на соединение с кнопками ✅/❌ |
| Каналы | Список активных каналов: ID, тип, количество блобов, время жизни |
| Отправить блоб | Выбор канала + текстовое поле + отправка (DHT → WS fallback) |
| DHT Хранилище | Ручная запись/чтение ключ-значение в DHT-сети |
| Таблица DHT | Визуализация корзин (buckets), непустые, с «мёртвыми» пирами |
| Лог | Поток событий: подключения, блобы, DHT-операции |

## Ключевое отличие от RobinHood

| Характеристика | RobinHood P2P | P2PPong v4.2 |
|---------------|---------------|--------------|
| Назначение | Мессенджер для пользователя | Стенд для разработчика |
| UI | Тематический (лес, бумага, утро) | Тёмный минимализм |
| Визуализация DHT | Нет (только счётчик пиров) | Полная таблица корзин |

## Сценарии использования

### Сценарий 1: Проверка ECDH-рукопожатия

**Цель:** убедиться, что два пира устанавливают общий секрет без раскрытия серверу.

**Шаги:**

1. Открыть P2PPong на Устройстве А.
2. Нажать «🏹 Создать маяк» → скопировать Peer ID из лога.
3. Открыть P2PPong на Устройстве Б.
4. Вставить Peer ID в поле «Peer ID друга...» → нажать «🤝 По Peer ID».
5. На Устройстве А появится заявка → нажать «✅».
6. В логе обоих устройств появится «✅ Подключено!» и «🔗 DHT: ...».

**Ожидаемый результат:** канал создан. Сервер видел только публичные ключи и хеш. Секрет канала известен только двум пирам.

**Как проверить:** сравнить ID канала на обоих устройствах. Они должны совпадать.

---

### Сценарий 2: Проверка шифрования блобов

**Цель:** убедиться, что сообщение расшифровывается только на целевом пире.

**Шаги:**

1. Выполнить Сценарий 1 (установить канал).
2. На Устройстве А выбрать канал в выпадающем списке.
3. Ввести сообщение → нажать «📤».
4. На Устройстве Б появится «📩 Блоб: [сообщение]...» в логе.

**Ожидаемый результат:** сообщение отображается только на Устройстве Б. В логе видно, что блоб ушёл либо через DHT, либо через WS (fallback).

**Как проверить:** в инструментах разработчика (F12 → Network → WS) посмотреть трафик WebSocket. Блоб представляет собой base64-строку фиксированной длины (4096 байт после паддинга). Содержимое нечитаемо.

---

### Сценарий 3: Наблюдение за DHT-сетью

**Цель:** увидеть, как пиры находят друг друга через Kademlia DHT.

**Шаги:**

1. Открыть P2PPong на трёх устройствах (А, Б, В).
2. На Устройстве А создать маяк.
3. На Устройстве Б подключиться к А.
4. На Устройстве В подключиться к А.
5. Смотреть на секцию «Таблица DHT» на всех трёх устройствах.

**Ожидаемый результат:**
- На каждом устройстве заполняются корзины (buckets).
- Видны ID пиров в корзинах.
- «Мёртвые» пиры (нет ответа > 120 сек) отображаются зачёркнутыми.
- Количество пиров в DHT (тег `DHT: N`) растёт.

**Как проверить:** закрыть вкладку Устройства Б. Через 120 секунд его ID станет зачёркнутым в таблице на Устройствах А и В.

---

### Сценарий 4: Проверка fallback-логики (DHT → WS)

**Цель:** убедиться, что сообщения доставляются даже при недоступности прямого DHT-канала.

**Шаги:**

1. Установить канал между двумя устройствами.
2. На Устройстве Б заблокировать WebRTC (например, через `chrome://webrtc-internals/` или файервол).
3. На Устройстве А отправить блоб.

**Ожидаемый результат:** блоб доставлен через WebSocket. В логе Устройства А будет видно, что DHT-пиров нет, fallback на WS.

**Как проверить:** в логе нет строки «DHT: ...» при отправке, но блоб получен на Устройстве Б.

---

### Сценарий 5: Ручная работа с DHT-хранилищем

**Цель:** понять, как данные распределяются по DHT-сети.

**Шаги:**

1. На Устройстве А ввести ключ (например, `test-key`) и значение (например, `hello-dht`).
2. Нажать «💾».
3. В логе появится «💾 DHT сохр: ... → N пиров».
4. На Устройстве Б ввести тот же ключ → нажать «🔍».

**Ожидаемый результат:** Устройство Б находит значение `hello-dht`.

**Как проверить:** если Устройство Б не подключено к DHT-сети (нет других пиров), поиск вернёт «❌ Не найдено». После подключения к сети — значение найдётся.

---

### Сценарий 6: Аудит метаданных на сигнальном сервере

**Цель:** понять, что видит и чего не видит сигнальный сервер.

**Шаги:**

1. Открыть инструменты разработчика (F12 → Network → WS).
2. Выполнить Сценарий 1 (установить канал).
3. Просмотреть все сообщения WebSocket.

**Что видит сервер:**
- `action: "subscribe"` — Peer ID подписывается на сообщения
- `action: "beacon"` — создан маяк (beaconId, tempKeyHash, peerId, pubKey)
- `action: "connect"` — запрос на подключение (targetPeerId, tempKeyHash, fromPeerId, pubKey, requestId)
- `action: "accept-request"` — принятие заявки (fromPeerId, tempKeyHash, myPeerId, pubKey)
- `action: "blob"` — зашифрованный блоб (channelId, blob: base64, peerId)
- `action: "dht-signal"` — SDP для WebRTC (targetPeerId, from, data)

**Что сервер НЕ видит:**
- Секреты каналов (они вычисляются через ECDH локально)
- Содержимое блобов (AES-256-GCM, ключ неизвестен серверу)
- Содержимое DHT-трафика (WebRTC DataChannel шифрован)

### Сигнальный сервер

- Сервер **технически способен** регистрировать метаданные: кто, когда и с кем устанавливал соединение.
- Сервер **видит** публичные ключи ECDH, что позволяет ему идентифицировать участников рукопожатия.
- Сервер **не видит** содержимого сообщений (блобов). Блобы зашифрованы AES-256-GCM ключом, который известен только двум пирам.
- Сервер **может** попытаться выполнить атаку «человек посередине» (MitM) на этапе рукопожатия, подменив публичные ключи. Защита от этого — внеполосная верификация Safety Number (в RobinHood) или ручное сравнение ключей (в P2PPong).

### DHT-сеть

- DHT-пиры видят IP-адреса друг друга (это необходимо для WebRTC DataChannel).
- DHT-пиры **не видят** содержимого блобов, если блоб зашифрован секретом канала.
- DHT-хранилище (`dht-store`) хранит данные **открытым текстом** внутри DHT-сети. Используйте его только для тестовых данных.
- «Мёртвые» пиры удаляются из корзин через 120 секунд, но их ID могут оставаться в логах.

### Блобы

- Размер блоба фиксирован (4096 байт) для предотвращения анализа трафика по размеру сообщения.
- Паддинг выполняется случайными байтами (через `crypto.getRandomValues`), а не пробелами или нулями.
- Несмотря на фиксированный размер, частота отправки блобов может быть использована для корреляции.

### WebRTC

- WebRTC DataChannel шифруется средствами браузера (DTLS). Это дополнительный уровень шифрования поверх AES-256-GCM.
- ICE-кандидаты могут раскрывать локальные IP-адреса пиров. Используйте VPN или Tor, если это неприемлемо.

### Идентификация

- Peer ID генерируется на основе характеристик устройства (WebGL, AudioContext, экран, память). Это **не** анонимный идентификатор.
- Соль хранится в localStorage. При очистке localStorage Peer ID изменится, и вы станете «новым пиром» для сети.
- Peer ID не привязан к личности, но может быть использован для долгосрочного отслеживания устройства.

### Ограничения браузера

- Приложение требует HTTPS для работы Web Crypto API.
- WebRTC может быть заблокирован корпоративными файерволами или некоторыми VPN.
- Вкладка должна быть активной для поддержания WebSocket-соединения. В фоновом режиме браузер может приостановить таймеры.

## Требования

- **HTTPS** (обязательно для Web Crypto API)
- Современный браузер с поддержкой WebRTC и WebSocket
